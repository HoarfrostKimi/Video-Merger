## 2026-07-06（晚）— 用户要求「运行项目」：启动受阻 + 根因 + 修复

- **现象**：`npm run dev` 一启动就崩：`TypeError: Cannot read properties of undefined (reading 'isPackaged')`，位置在 `out/main/index.js` 的 `electron.app.isPackaged`。
- **根因（两层）**：
  1. 代码层：`@electron-toolkit/utils` 被 electron-vite 内联进主包，其模块顶层 `const is = { dev: !electron.app.isPackaged }` 在 `electron.app` 就绪前急于求值 → 崩溃。修复：移除该依赖导入，改用已具名导入的 `app.isPackaged`（即 `const dev = !app.isPackaged`）、`app.setAppUserModelId`；删掉无边框窗口才需要的 `optimizer.watchWindowShortcuts`。
  2. 环境层（更根本）：本沙箱**无法启动 Electron 的 GUI 运行时**——`electron -e "require('electron')"` 返回的是 **string** 而非对象，说明 electron 未以真正 App 主进程启动（`electron.app` 永远 undefined），且无显示设备。所以窗口在本环境根本渲染不出来，与代码无关。
- **验证**：typecheck(`tsc -b`) 0 error；`npm run build` 成功；`npx vitest run` 85 passed / 0 error（修复中曾因测试 mock 缺 `app.setAppUserModelId` 产生 3 个 unhandled rejection，用 `app.setAppUserModelId?.()` 可选链消除）。
- **提交**：`e75bb12`（仅改 `src/main/index.ts`，7 增 8 删），走 pre-commit 质量门（生成 `.quality/*.marker` 后自动清理）。
- **结论**：本沙箱只能验证「代码可编译/可构建/测试通过」，无法真正渲染 GUI；用户在**自有桌面（Windows+显示器）**执行 `npm run dev` 或 `npm run build && npx electron-builder` 即可正常运行。
- **可复用诊断法（修正）**：`electron -e "typeof require('electron')"` 返回 string 是**正常现象**（普通 node 进程里 require('electron') 本就返回路径字符串），**不能**据此判定环境无法启动 GUI。正确判据是用 electron 二进制直接跑探针：`./node_modules/electron/dist/electron.exe probe.js`，其中 probe 检查 `typeof require('electron').app` 与 `app.whenReady` 是否可用。

## 2026-07-06（晚，二次）— 用户再次要求「运行项目」：找到真正根因

- **环境实证（决定性）**：用 `electron.exe` 直接跑最小探针，输出 `TOP-LEVEL app=undefined ipcMain=undefined BrowserWindow=undefined`，且 `app.whenReady` 不可用。结论：**本沙箱里 electron 主进程核心模块（app/ipcMain/BrowserWindow）根本未初始化**，连 ready 之后都无法用。这是**纯环境限制**——任何 Electron 应用（含 hello world）在此都跑不起来，与项目代码无关。
- **前次 e75bb12 的误判**：当时把 `const dev = !app.isPackaged` 留在模块顶层，以为修好了；实际只把崩溃点从 `@electron-toolkit/utils` 包内部挪到自己顶层代码，dev 启动时仍崩在 isPackaged（被截断输出误导）。
- **真正稳健修复**：把 `dev = !app.isPackaged` 从模块顶层移入 `createWindow()` 内部（whenReady 之后才执行）。这样模块加载阶段不再访问任何 electron 成员。提交 `69b9cb4`。
- **说明**：该修复对**桌面环境**是有价值的稳健性改进（避免模块加载期访问可能未就绪的 app），且原 safeHandle 顶层注册 ipcMain 在桌面环境可用故不影响运行；但**在本沙箱仍跑不了**（因环境里 ipcMain 也是 undefined，根因是环境不初始化 electron 主进程，非代码）。
- **验证**：vitest 85 passed；tsc -b 0 error；npm run build 成功（renderer 2MB / main 23.95kB / preload 1.51kB）。
- **最终结论**：本沙箱只能验证「编译/构建/测试全绿」，**无法渲染 Electron GUI**；用户在自有 Windows 桌面执行 `npm run dev` 即可正常运行（现代码已修到能启动）。
