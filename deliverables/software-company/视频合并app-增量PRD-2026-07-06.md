## 2. 产品定义

### 2.1 Product Goals（3 个清晰、正交的目标）

1. **安全稳健**：满足 Electron 安全清单（sandbox + CSP + 最小权限 IPC），并修复 `userData` 误置安装/项目目录导致的正式版配置持久化失效风险。
2. **响应与可观测**：消除主进程同步扫描阻塞，使大目录不卡 UI；进度估算基于真实时长，用户能看到平滑、准确的中间进度。
3. **可维护与可验证**：开启 `strict`、消除 `any` 与魔法值、收敛重复声明、修复 Promise 反模式，并使单元测试真正引用真实源码，从而获得真实覆盖与回归防护。

### 2.2 User Stories（以「开发者 / 维护者」为验收视角）

- As a **维护者**，I want 启用 sandbox 并加 CSP、对 IPC 入参做校验，so that 应用符合 Electron 安全清单、攻击面最小化。
- As a **维护者**，I want `userData` 回归 Electron 默认路径，so that 安装到 `Program Files` 等只读位置时配置不会静默写入失败。
- As a **维护者**，I want 扫描目录改为异步，so that 递归遍历数千文件时不阻塞主进程事件循环、IPC 不卡顿。
- As a **维护者**，I want 进度基于各文件真实探测时长求和，so that 用户在合并过程中看到平滑且可信的中间百分比（而非 0→100 跳变）。
- As a **维护者**，I want 测试文件 import 真实源码并 mock ffmpeg，so that 源码改动引入的 bug 能被测试捕获，而非只验证副本。
- As a **维护者**，I want 开启 `strict` 并消除 `any` / 魔法值 / 重复类型，so that 类型约束在编译期生效，长期维护成本下降。

## 3. 技术规范

### 3.1 Requirements Pool

> 优先级：**P0 = 必须（本次交付底线）** / **P1 = 应做** / **P2 = 可选（仅评估或后续）**
> 标签：`[A]` 安全 / `[B]` 性能 / `[C]` 代码质量；括号标注对应评审项与源码改动点。

#### P0 — Must have

| 编号 | 需求 | 验收标准 | 改动点 |
|------|------|----------|--------|
| A1 | 启用 `sandbox: true`（若可行） | 渲染进程在 sandbox 下正常加载、preload 行为不变；ffmpeg 子进程 spawn 不受影响（见 Open Q1） | `src/main/index.ts:57-60` |
| A2 | 增加 Content-Security-Policy | 通过 `session.defaultSession` 或 `<meta http-equiv>` 设置 CSP，禁止非必要 remote / inline 脚本；本地内容无外部源 | 主进程 `BrowserWindow` 创建处 / `index.html` |
| A3 | `userData` 回归 Electron 默认路径 | 移除 `app.setPath('userData', join(__dirname,'../../user-data'))`；配置读写落到系统默认 `userData` | `src/main/index.ts:356-357` |
| A4 | IPC 入参（folderPath / outputPath / filePaths）合法性与越界校验 | 对路径做非空、存在性、绝对路径、越界（如禁止指向敏感系统目录）校验；非法入参返回明确错误而非静默执行 | `src/main/index.ts:107,322,337` 等 handler |
| C1 | 开启 `strict` 并处理因此暴露的类型问题 | `tsconfig.node.json` / `tsconfig.web.json` 开启 `"strict": true`；`tsc` 通过（或按 Open Q3 分阶段） | 两个 tsconfig + 全部源码 |
| C2 | 消除 `any` | preload `invokeApi` 返回类型与 `env.d.ts` 对齐；IPC handler 入参去除裸 `any` | `src/preload/index.ts:9-18`、`src/main/index.ts:283,316,330,345` 等 |
| C6 | 修复 Promise 反模式 | `mergeVideos` 的 `new Promise(async (resolve,reject)=>{...})` 改为 `new Promise((resolve,reject)=>{... async function run(){...}; run().catch(reject) })` 或 `async` 函数 + 内部 `Promise` 封装，确保 async 执行器内异常能触发 `reject` | `src/main/ffmpeg.ts:92` |

### 3.2 UI Design Draft

**无新增 UI，仅配置 / 逻辑层改动。**

- 所有改动位于主进程（`src/main/*`）、preload（`src/preload/*`）、tsconfig、依赖与测试层；渲染进程 UI（窗口、按钮、表格、进度条）外观与交互**保持不变**。
- 用户可感知的间接变化（不新增界面元素）：
  - 配置持久化位置随 A3 变更（旧配置迁移见 Open Q2），UI 上「记住输入/输出目录」功能行为不变。
  - 进度条数值随 B2 更准确平滑（仍走现有 300ms 轮询通道，轮询间隔本身可借 C3 提取为常量但默认不改）。
- 若 A1 启用 sandbox 导致某渲染能力异常，属于回归风险，需在 Open Q1 决策中处理，不引入新 UI 兜底。

### 3.3 Open Questions（待确认问题 · 需用户 / 架构师拍板）

1. **【安全·sandbox】** 启用 `sandbox: true` 是否会破坏 preload 中 ffmpeg 子进程的 `spawn`？当前 preload 通过 `electron-toolkit` 的 `electronAPI` 暴露了**完整** `ipcRenderer`（含 `invoke/send/on`）。若 sandbox 启用后 spawn 受限或行为异常，是否退而求其次——**仅做 CSP + IPC 入参校验 + 收敛 `electronAPI` 暴露**，暂缓 `sandbox:true`？
2. **【配置迁移】** `userData` 回归默认路径后，`config.json` 读取位置从「项目/安装目录 `user-data`」变为系统默认 `userData`（如 `%APPDATA%/video-merger`）。**旧用户配置如何迁移 / 兼容**？开发期是否仍允许通过环境变量或命令行参数指向项目内目录以便调试？
3. **【strict 范围】** 开启 `strict` 可能引爆大量历史类型错误（现有 tsconfig 未开 `strict`）。是否**在本次一次性修完**，还是**分阶段**（先开 `strict` 并临时用 `// @ts-expect-error` 或 `strictNullChecks` 等子集灰度）？需据此确定 C1 的交付边界与工期。
4. **【并行合并范围】** B3（多组合并行）默认仅评估不实现；若希望本次一并落地，需确认并发上限与磁盘/CPU 资源预算（见 B3 风险）。
5. **【electronAPI 暴露】** 是否随本次安全加固一并移除 preload 中完整 `electronAPI` 暴露（仅保留受限 `api`）？移除前需确认渲染进程无其他代码依赖原始 `ipcRenderer`。
