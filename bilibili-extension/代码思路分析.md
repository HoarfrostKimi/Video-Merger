# B站投稿助手 —— 代码思路分析

## 一、项目定位
一个 **Chrome MV3 扩展**，在 B站投稿页注入「助手面板」，自动上传本地（或视频合并 app 生成的）MP4 并提交稿件。
它不是独立工具，而是「视频合并 app」的**下游执行端**：合并 app 在本地起 HTTP 服务，把视频 URL 传给扩展，扩展操控 B站页面完成投稿，再反向通知 app 关闭。

## 二、文件结构
| 文件 | 作用 |
|------|------|
| `manifest.json` | MV3 配置。`content_scripts` 注入到 `member.bilibili.com/platform/upload/video/frame`；`host_permissions` 放行 `127.0.0.1`/`localhost`（与 app 联动用） |
| `content.js`（核心 ~920 行） | 工具函数 + 6 步投稿 + 批量上传 + 注入面板 UI + app 联动 |
| `content.css` | 面板浮窗样式 |
| `popup.html` / `popup.js` | 扩展图标弹窗，仅一个「打开投稿页」按钮 |

## 三、核心设计思路

### 1. 注入式 UI
在投稿页 DOM 注入固定浮窗 `#bili-helper-panel`（右上角）。用 `getElementById` 判断防重复注入。
面板含：可见范围下拉、选择 MP4 文件、文件预览、开始按钮、实时日志区。

### 2. DOM 驱动的自动化（最关键的一点）
B站投稿页是 **Vue SPA，无公开 API**。扩展的思路 = **模拟用户操作 DOM**：
- **文件注入**：用 `DataTransfer` 构造文件列表，再用原生 setter
  `Object.getOwnPropertyDescriptor(HTMLInputElement.prototype,'files').set`
  绕过只读的 `files` 属性，最后 `dispatchEvent('change'/'input')` 让 Vue 感知到文件。这是处理受控 input 的经典手法。
- **按钮点击**：对关键按钮（如封面「完成」）用
  `dispatchEvent(new MouseEvent('click',{clientX,clientY,bubbles}))`
  模拟真实坐标点击，避免 Vue 不响应普通 `.click()`。
- **元素定位（多策略回退）**：
  1. CSS selector（`.el-select`、`.check-radio-v2-container` 等）
  2. 文本匹配（`findButtonByText` / `waitForElementByText`）
  3. 向上找可点击父元素（`toClickable`：`button`/`a`/`role=button`/class 含 btn/`cursor:pointer`）
  - 每种定位都准备了「主方案 + 备用方案」，失败不崩溃，只打日志提示手动处理。

### 3. 六步投稿流程 `uploadAndSubmit`
1. **上传文件**：注入到 `file input`
2. **等待表单**：不等上传完成（B站支持边传边填），用 `MutationObserver` 等标题输入框出现
3. **创作声明**：设为「内容无需标注」，按「创作声明」标签文本定位 `el-select`
4. **可见范围**：操作 `check-radio-v2` 自定义组件（非原生 radio/select）
5. **封面**：首个视频用系统推荐（直接点完成）；其余从视频截帧作为封面上传
6. **立即投稿**：点击提交按钮

### 4. 封面截帧 `captureFrameFromFile`（纯前端）
`URL.createObjectURL` 把 File 变 `<video>` → `seek` 到第 2 秒 → `drawImage` 到 canvas → 按 **4:3 居中裁剪** → `toBlob` 成 jpeg。

### 5. 批量上传 `batchUpload`（一个稿件多分 P）
- **第 1 个视频**：走主流程（主 input 上传 → 填表 → 提交）
- **第 2+ 个视频**：点「添加分P」→ 在 `.bcc-upload-wrapper` 内找 input 注入文件 → 等卡片出现 → 点卡片进编辑页 → 再走填表提交
- 用 `MutationObserver` 替代 `setInterval` 轮询；单视频失败 `try/catch` 不中断整体批处理。

### 6. 与合并 app 的联动（自动上传）
- 扩展支持 URL 参数 `?autoFiles=视频URL列表` 自动触发。
- 检测到参数后：
  1. 立即 `history.replaceState` 清掉参数（**防止刷新页面重复投稿**）
  2. `fetch` 下载每个视频 blob → 转成 `File`
  3. 跑 `batchUpload`
  4. 完成后 `fetch('?signal=done')` 通知 app 关闭
- 用**本地 HTTP 服务器做桥**：扩展无法直接读 app 文件，只能传 URL。

### 7. 容错与可观测
- 全程 `onLog` 把进度实时写到面板日志。
- 关键等待用 `waitForElement`（MutationObserver + 超时），而非纯 `sleep`（但仍有多处 sleep 兜底）。
- 每步都有 `try/catch` + 「提示：未找到…请手动」兜底。
- 可见范围记忆到 `localStorage`。

## 四、设计亮点
- **纯前端、零 B站 API 依赖**：登录态 cookie 由浏览器自动携带。
- **多策略定位 + 回退**，对抗 B站 DOM 频繁变化。
- **MutationObserver 异步等待**，比轮询更省资源。
- **本地 HTTP 桥接**解耦了 app 文件系统与扩展的访问限制。

## 五、可改进点
- 标题/简介/标签**没有编辑 UI**，仅用文件名正则解析标题。
- 多处**固定 `sleep`（2s/5s）**，B站页面慢时可能不稳。
- **未处理 B站二次确认/验证码弹窗**（如风控验证）。
- `findButtonByText` 全文档 `querySelectorAll('*')` 扫描，视频多时性能一般。
- `host_permissions` 仅 `127.0.0.1`/`localhost`，若合并 app 换端口需同步改 manifest。
